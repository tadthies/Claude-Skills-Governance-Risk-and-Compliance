WCAG Accessibility Issues in ProfileModal Component
====================================================

The following WCAG 2.1 accessibility issues were identified in the ProfileModal component:

---

1. MISSING ARIA ROLE FOR DIALOG (WCAG 4.1.2 - Name, Role, Value)
   The modal container div lacks role="dialog" (or role="alertdialog"). Without this,
   assistive technologies like screen readers will not announce this as a dialog,
   and users will have no indication they are in a modal context.
   Fix: Add role="dialog" and aria-modal="true" to the modal-content div.

2. MISSING ACCESSIBLE NAME FOR THE DIALOG (WCAG 4.1.2 - Name, Role, Value)
   The dialog has no aria-labelledby or aria-label attribute to provide an accessible
   name. Screen readers need to announce the purpose of the dialog when it receives
   focus.
   Fix: Add aria-labelledby pointing to the h2 element's id
   (e.g., <h2 id="modal-title">Edit Profile</h2> and aria-labelledby="modal-title").

3. NO FOCUS MANAGEMENT / FOCUS TRAP (WCAG 2.1.2 - No Keyboard Trap; 2.4.3 - Focus Order)
   When the modal opens, focus is not programmatically moved into the modal. Users
   navigating by keyboard will remain focused on whatever element was active before,
   making it difficult or impossible to interact with the modal. Additionally, there
   is no focus trap, so Tab/Shift+Tab can cycle focus outside the modal while it is
   open, which is disorienting.
   Fix: On open, use useEffect and useRef to move focus to the first focusable element
   (or the modal container itself). Implement a focus trap that constrains keyboard
   navigation within the modal while it is open.

4. MISSING RETURN OF FOCUS ON CLOSE (WCAG 2.4.3 - Focus Order)
   When the modal closes, focus is not returned to the element that triggered it
   (the "Edit Profile" button). Screen reader and keyboard users will lose their
   place in the page.
   Fix: Store a reference to the trigger element before opening the modal and restore
   focus to it when the modal closes.

5. NO KEYBOARD HANDLER FOR ESCAPE KEY (WCAG 2.1.1 - Keyboard)
   The modal provides no way to close it via the keyboard (specifically the Escape
   key, which is the standard pattern for dismissing dialogs per ARIA Authoring
   Practices Guide). Keyboard-only users cannot dismiss the modal without clicking
   outside it or pressing Cancel.
   Fix: Add a keydown event listener (e.g., onKeyDown on the dialog) that calls
   onClose when the Escape key is pressed.

6. CLICK-TO-CLOSE OVERLAY HAS NO KEYBOARD EQUIVALENT (WCAG 2.1.1 - Keyboard)
   The modal-overlay div uses onClick={onClose} to close the modal when users click
   outside the content area. This is a mouse-only interaction; keyboard users cannot
   replicate this behavior by tabbing to the overlay. Divs are not natively focusable
   or activatable.
   Fix: Either remove this pattern and rely on an explicit close button + Escape key,
   or add role="button", tabIndex={0}, and a corresponding onKeyDown handler to make
   the overlay keyboard-accessible.

7. FORM INPUTS USE <span> INSTEAD OF <label> (WCAG 1.3.1 - Info and Relationships; 1.3.5 - Identify Input Purpose; 2.4.6 - Headings and Labels)
   The "Full Name" and "Email" field labels are plain <span> elements, which are not
   programmatically associated with their corresponding inputs. Screen readers will
   not announce the label when the input is focused, leaving users without context
   for what to type.
   Fix: Replace <span> with <label> and use either a for/htmlFor attribute matching
   each input's id, or wrap the input inside the label element.

8. PLACEHOLDER TEXT USED AS THE ONLY LABEL ALTERNATIVE (WCAG 1.3.1; 3.3.2 - Labels or Instructions)
   While placeholders are present ("Enter your name", "Enter email"), placeholder
   text disappears as soon as users begin typing and has insufficient color contrast
   in many browsers by default. Placeholders must never be used as a substitute for
   visible, persistent labels.
   Fix: Keep labels (see issue 7) as the primary label; treat placeholder as optional
   hint text only.

9. NO VISIBLE CLOSE BUTTON (WCAG 2.4.3 - Focus Order; 2.1.1 - Keyboard; ARIA Dialog Pattern)
   There is no dedicated close/dismiss button (typically an "X" button) in the modal
   header. The only way to close the modal is the Cancel button or clicking outside.
   ARIA Authoring Practices recommend a visible close button within the dialog.
   Fix: Add a close button (e.g., an X icon button) in the modal header with an
   accessible name like aria-label="Close dialog".

10. INSUFFICIENT COLOR CONTRAST CANNOT BE VERIFIED WITHOUT STYLES (WCAG 1.4.3 - Contrast Minimum)
    The code provides no color information. If the chosen text colors, background
    colors, or placeholder colors do not meet the 4.5:1 ratio (normal text) or 3:1
    (large text / UI components), this will be a WCAG failure.
    Fix: Audit CSS for all text/background color pairings and ensure they meet minimum
    contrast ratios. Placeholder text typically needs special attention.

11. FORM HAS NO SUBMIT HANDLER / ERROR HANDLING (WCAG 3.3.1 - Error Identification; 3.3.3 - Error Suggestion)
    The form lacks onSubmit handling and provides no visible or programmatic error
    messages if validation fails. Users with screen readers will not be informed of
    any errors after attempting to submit.
    Fix: Add onSubmit, perform validation, and surface error messages using
    aria-describedby or aria-live regions so that screen readers announce errors.
